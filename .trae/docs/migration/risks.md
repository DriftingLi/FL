## R1: API 契约不一致

**触发条件**：Go 后端某个端点的响应字段名、嵌套结构、分页格式或错误码与 Python 版不一致，导致前端解析失败。

**影响**：前端功能不可用，用户体验受损。

**应对策略**：
1. ✅ 已逐一对照 14 个 Python 蓝图，在 Go 中等价实现（路径/方法/请求体/响应体）
2. ✅ 保持响应统一结构 `{code, message, data}`
3. ⏳ **待执行**：Task 37 集成测试中的双跑对比（同请求同时打到 Python 与 Go，断言响应等价）
4. ⏳ **待执行**：前端切换 API base URL 到 staging，全功能回归测试
5. **回退方案**：如发现不一致，立即修复 Go 实现；严重时切回 Python 后端

**负责人**：后端开发

---

## R2: 数据迁移丢失或损坏

**触发条件**：PostgreSQL→PostgreSQL 数据迁移过程中（从 Python 版 schema 迁移至 Go 版 schema），行数不匹配、字段类型映射错误、JSON 数据损坏或外键关系断裂。

**影响**：学员数据、课程内容、考试记录丢失，无法恢复。

**应对策略**：
1. ✅ 迁移工具按外键依赖顺序处理 19 张表，分批 commit（每 1000 行）
2. ✅ 使用 `OVERRIDING SYSTEM VALUE` 保留原始主键
3. ✅ 迁移后自动执行行数校验（源 vs 目标）
4. ✅ 迁移后重置 IDENTITY 序列，防止后续插入主键冲突
5. ✅ 支持 `--dry-run` 模式，仅校验行数不写入
6. ⏳ **待执行**：在 staging 环境用生产快照演练，记录耗时
7. **安全措施**：迁移前对源 PostgreSQL 做完整备份；迁移期间标记只读

**负责人**：DBA / 后端开发

## R3: 性能回归

**触发条件**：Go 后端在核心接口（登录、课程列表、练习提交、考试提交）的 P95 延迟或吞吐量未达预期（P95 降 30%+，吞吐 3 倍+）。

**影响**：用户体验未改善，重构投入产出比低。

**应对策略**：
1. ⏳ **待执行**：Task 38 使用 vegeta/k6 压测核心接口
2. ⏳ **待执行**：对比 Python 版本性能基线
3. **优化方向**：
   - 连接池配置（已设置 MaxOpenConns=20, MaxIdleConns=5）
   - N+1 查询排查（GORM Preload vs 显式查询）
   - JSON 序列化优化（sonic vs encoding/json）
4. **可接受范围**：若 P95 持平但吞吐提升 2 倍+，仍可接受

**负责人**：后端开发

## R4: AI/文件依赖不可用

**触发条件**：
- 智谱 GLM API 不可用或响应格式变化
- LibreOffice 未安装或版本不兼容，导致 PPT→PDF 转换失败
- Coze OAuth 私钥配置错误

**影响**：AI 评分、文本生成、课件转换功能不可用。

**应对策略**：
1. ✅ AI 评分实现多级降级 fallback（智谱 → OpenAI → 默认评分）
2. ✅ AI 连接测试端点 `/api/ai/test` 可快速诊断
3. ✅ LibreOffice 通过 Docker sidecar 容器提供，无宿主污染
4. ✅ Coze OAuth 未配置时返回明确错误提示
5. **监控**：AI 调用失败率超 10% 时告警
6. **降级**：AI 不可用时，简答题可手动阅卷；文件转换失败时保留原始文件

**负责人**：后端开发

## R5: JWT 令牌不兼容

**触发条件**：Go 版 JWT claims 结构（`user_id`/`username`/`role`）与 Python 版不一致，或签名算法/密钥不同，导致存量令牌无法在新后端通过校验。

**影响**：所有已登录用户在切换瞬间被强制登出。

**应对策略**：
1. ✅ claims 结构完全对齐（`user_id`/`username`/`role`，1 小时过期）
2. ✅ 使用相同 JWT 密钥（`JWT_SECRET_KEY` 环境变量）
3. ✅ 签名算法一致（HS256）
4. **过渡方案**：切换前将 JWT 过期时间设为较短（如 15 分钟），确保存量令牌快速过期；切换后用户重新登录获取新令牌
5. **验证**：Task 37 集成测试中验证 Python 生成的令牌在 Go 后端可用

**负责人**：后端开发

## R6: 回滚困难

**触发条件**：Go 后端上线后发现严重问题，需要切回 Python 后端，但数据库 schema 已变（Go 版 DDL 替代 Python 版 SQLAlchemy 自动生成的 schema），无法直接回退。

**影响**：服务不可用时间延长。

**应对策略**：
1. ✅ 过渡期 Python 与 Go 后端并行运行（不同端口/域名）
2. ✅ 数据迁移为 PostgreSQL→PostgreSQL 单向复制（Python 版 schema → Go 版 schema）；回滚时 Python 仍连原 PostgreSQL（数据可能滞后）
3. **蓝绿部署**：Go 后端先部署到 staging，验证通过后再切生产流量
4. **灰度方案**：先切 10% 流量到 Go，观察 24 小时无异常后全量切换
5. **快速回滚**：保留 Python 后端镜像至少 7 天，可通过 Railway 一键切回
6. **数据回滚**：如 Go 版 PostgreSQL 数据被污染，从迁移前的 PostgreSQL 备份恢复

**负责人**：运维 / 后端开发

## R7: 部署配置错误

**触发条件**：Dockerfile 构建失败、Railway 配置错误、环境变量缺失、健康检查不通过。

**影响**：部署失败或服务无法启动。

**应对策略**：
1. ✅ Dockerfile 多阶段构建，已验证 `go build` 通过
2. ✅ railway.toml 配置健康检查路径 `/api/health`
3. ✅ .env.production 模板覆盖全部必填变量
4. ✅ 生产环境配置校验（缺失 SECRET_KEY/JWT_SECRET_KEY/DATABASE_URL 时启动失败）
5. **验证**：Task 41 staging 环境全流程验证

**负责人**：运维

## R8: 并发/事务行为差异

**触发条件**：Python Flask-SQLAlchemy 与 Go GORM 的事务隔离级别、锁行为、并发控制不一致，导致数据竞争或脏读。

**影响**：考试评分错误、学员等级异常、数据不一致。

**应对策略**：
1. ✅ GORM 默认事务封装写操作
2. ✅ 阅卷评分使用显式事务（`updateParticipantScore`）
3. ⏳ **待验证**：Task 36 单元测试覆盖 `-race` 标志
4. ⏳ **待验证**：Task 37 集成测试跑通并发场景
5. **关键路径**：考试交卷、等级晋级、错题入库需重点测试并发安全

**负责人**：后端开发
