# 叉车维修培训 + 残值评估 项目合并计划

## 目标
将 `d:\叉车维修项目\残值评估` 作为「残值评估模块」合并到 `d:\叉车维修项目\维修培训` 中，最终形成单一仓库；合并完成后初始化 git 并撰写 README.md。

## 用户决策
- **Git**：合并完成后再 `git init`（不保留合并前中间状态）
- **后端**：完全合并到 `backend-go`（单一 Go module、单一进程、单一端口）
- **前端**：统一为 **TypeScript**。两个方向同时进行：
  1. 将维修培训前端从 JS 升级到 TS（启用 tsconfig、改 main.js→main.ts、所有 .vue 加 `lang="ts"`、所有 src/*.js 改为 *.ts 并补类型）
  2. 将残值评估前端从 Ant Design Vue 改写为 Element Plus（保持 TS 不变）
  最终两个前端合并为单一 TypeScript + Element Plus 项目。
- **结构**：合并后清理文件夹嵌套（把 `维修培训/` 子目录内容上移到项目根，删除 `残值评估/` 与空的 `维修培训/`）

## 实施步骤

### 阶段 1：后端合并（核心，最先做）

#### 1.1 创建 valuation 子包目录
在 `维修培训/backend-go/internal/valuation/` 下创建子目录：`config/`、`handler/`、`model/`、`repository/`、`service/`、`migration/`。

#### 1.2 搬移并改造残值评估后端代码
将 `残值评估/backend/internal/` 下对应目录的 `.go` 文件搬到 `backend-go/internal/valuation/` 对应子目录：

- `internal/handler/*.go` → `internal/valuation/handler/*.go`
- `internal/model/*.go` → `internal/valuation/model/*.go`
- `internal/repository/*.go` → `internal/valuation/repository/*.go`
- `internal/service/*.go` → `internal/valuation/service/*.go`
- `internal/migration/migration.go` → `internal/valuation/migration/migration.go`
- `internal/migration/sql/*.sql` → `internal/valuation/migration/sql/*.sql`
- `internal/config/{loader,logger,database}.go` → `internal/valuation/config/*.go`
- `pkg/pdf/*.go` → `backend-go/pkg/pdf/*.go`

**模块路径替换**：所有 `forklift-backend/internal/...` import 改为 `forklift-training/internal/valuation/...`；`forklift-backend/pkg/pdf` 改为 `forklift-training/pkg/pdf`。

**测试文件**：保留 `*_test.go`（battery_rul_test、valuation_test、kcondition_test、khours_test、ktime_test、kothers_test、helpers_test、integration_test、template_test），同样改 import 路径。

#### 1.3 合并 go.mod / go.sum
- 升级 Go 版本到 `1.26.4`（残值评估所需；维修培训代码兼容）
- 合并 require 块：新增 `github.com/jackc/pgx/v5`、`github.com/jung-kurt/gofpdf`、`github.com/spf13/viper`、`go.uber.org/zap`、`go.mongodb.org/mongo-driver/v2`（间接）等
- 跑 `go mod tidy` 自动整理 go.sum

#### 1.4 合并数据库迁移
- 保留 `migrations/000001_init.{up,down}.sql`（维修培训原 19 表）
- 新建 `migrations/000002_valuation.up.sql`：把残值评估 `000001_init.up.sql` + `000003_battery_rul.up.sql` 内容合并（6 + 1 = 7 张表）
- 新建 `migrations/000002_valuation.down.sql`：DROP 这 7 张表
- 新建 `migrations/000003_valuation_seed.up.sql`：把残值评估 `000004_seed.up.sql` + `000005_seed_repair.up.sql` 合并
- 新建 `migrations/000003_valuation_seed.down.sql`
- 新建 `migrations/000004_brand_models_repair.up.sql` / `.down.sql`：从残值评估 `000006` 搬入
- 调整 `internal/valuation/migration/migration.go`：迁移文件目录指向 `backend-go/migrations/`，迁移文件名前缀改为新的 `000002/000003/000004`

**关键决策**：维修培训与残值评估共用同一个 `DATABASE_URL`（同一 PostgreSQL 实例和库）。表名无冲突，无需 schema 隔离。

#### 1.5 扩展配置（`.env` + `config.go`）
- 在 `backend-go/.env.example` 新增字段：
  - `VALUATION_PDF_OUTPUT_DIR=storage/reports`
  - `VALUATION_LOG_LEVEL=info`
  - `VALUATION_LOG_FORMAT=console`
  - 复用 `DATABASE_URL`（不另设）
- 在 `internal/config/config.go` 的 `Config` 结构体新增 `Valuation` 子结构：
  ```go
  type ValuationConfig struct {
      PDFOutputDir string
      LogLevel     string
      LogFormat    string
      CORSOrigins  []string  // 复用顶层
  }
  ```
- 修改 `internal/valuation/config/loader.go`：从环境变量加载（不再读 yaml），或保留 viper 但默认从环境变量读
- 删除 `backend-go/config/config.yaml` 依赖（统一走 .env）

#### 1.6 修改主入口 `cmd/server/main.go`
在现有 main.go 中按顺序追加：
1. 加载 valuation 配置（从 cfg.Valuation 读取）
2. 创建 pgx 连接池（与 GORM 共用 `cfg.DatabaseURL`，pgx 解析同样的 postgres URL）
3. 运行 valuation 迁移（仅当 `cfg.DatabaseURL` 是 postgres 时；sqlite 时跳过）
4. 装配 valuation 服务（coefficientLoader、brandLoader、partConfigLoader、valuationSvc、batterySvc）
5. 装配 PDF 生成器
6. 在 `api.NewRouter(...)` 之后调用 `valuation.RegisterRoutes(r, ...)` 注册残值评估路由

**路由前缀决策**：所有残值评估路由统一挂在 `/api/valuation/*` 下（避免与维修培训 `/api/*` 冲突，也避免使用 `/api/v1/*` 引起混淆）：
- `POST /api/valuation/evaluations`
- `GET  /api/valuation/evaluations`
- `GET  /api/valuation/evaluations/:id`
- `POST /api/valuation/evaluations/:id/report`
- `GET  /api/valuation/evaluations/:id/report`
- `GET  /api/valuation/part-configs`
- `GET  /api/valuation/brands`
- `GET  /api/valuation/coefficients`
- `PUT  /api/valuation/coefficients/:key`
- `POST /api/valuation/historical-sales/import`
- `POST /api/valuation/battery/evaluations`
- `GET  /api/valuation/battery/evaluations`
- `GET  /api/valuation/battery/evaluations/:id`
- `POST /api/valuation/battery/evaluations/:id/report`
- `GET  /api/valuation/battery/evaluations/:id/report`

新建 `internal/valuation/router.go`（参考 `internal/api/router.go` 模式）暴露 `RegisterRoutes(r *gin.Engine, ...)`。

#### 1.7 中间件复用
- 复用维修培训的 `middleware.RequestID/Logger/Recovery/CORS`
- 残值评估原有 `middleware.go`（LoggerMiddleware/RecoveryMiddleware/CORSMiddleware）**不搬入**，避免重复
- valuation 路由组可选加 `middleware.JWTAuth(cfg)`（让 valuation 模块也受登录保护，符合"作为维修培训模块"的定位；但若用户后续要求开放访问可去掉）

**决策**：默认对 valuation 路由也启用 JWTAuth，与维修培训一致；学员登录后才能用残值评估功能。

#### 1.8 删除 embed 静态托管
- 不搬入 `残值评估/backend/cmd/server/static.go`（embed.FS 模式）
- 维修培训后端继续以本地 `static/` 目录方式提供静态资源，前端独立部署/构建
- 后端不嵌入前端 dist

#### 1.9 验证后端
```powershell

go build ./...
go test ./...
```

### 阶段 2：前端合并（统一 TypeScript + Element Plus）

#### 2.A 维修培训前端 JS → TS 升级

##### 2.A.1 添加 TS 工具链
在 `维修培训/frontend/package.json` 的 devDependencies 中新增：
- `typescript ^5.7.3`
- `vue-tsc ^2.1.10`
- `@types/node ^22.10.5`

build 脚本改为：`"build": "vue-tsc --noEmit && vite build"`

##### 2.A.2 添加 tsconfig
将 `残值评估/frontend/tsconfig.json` 与 `tsconfig.node.json` 复制到 `维修培训/frontend/`，内容基本保持一致（`paths: { "@/*": ["src/*"] }` 已匹配现有 alias）。

## 风险与备选

| 风险 | 影响 | 缓解 |
|------|------|------|
| Go 1.26 升级后维修培训代码可能有小兼容问题 | 中 | 升级后跑全部测试；go vet 检查 |
| sqlc 生成的 repository 代码可能与 pgx v5 版本耦合 | 中 | 保留 sqlc.yaml 与 queries/，如有需要重新生成 |
| 维修培训 JS→TS 升级涉及约 50+ 文件（15 个 api + 3 个 store + 8 个 util + 6 个 composable + 50+ 个 .vue） | 高 | 渐进式：tsconfig 先开 `allowJs: true`、`strict: false`，先保证构建通过；类型补全分批进行，函数体内允许 `any` 占位 |
| Ant→Element 转换工作量大（约 7 个 view + 7 个 component） | 高 | 严格按转换表机械转换；保留原 TS 文件作为参考直到验证完成 |
| echarts 6 vs 5 API 差异 | 低 | 主要 API 兼容；按需调整 |
| pgxpool 与 GORM 共用同一 DB 时连接池配置 | 低 | 各自独立连接池，互不影响 |
| 路径相对引用在 `os.Executable()` 模式下仍能工作 | 低 | 已确认 `resolveStaticDir` 上溯两级，移动目录后仍生效 |
| 静态资源 `/static/*` 路径与未来前端构建产物路径冲突 | 低 | 维修培训后端不嵌入 dist，前端独立部署，无冲突 |
| vue-tsc 严格类型检查可能阻塞 build | 中 | 初期 `strict: false`；type-check 与 build 解耦：build 仅 `vite build`，type-check 单独 npm script（可选执行） |
